# ML Pipeline Audit — 2026-04-20

**Scope:** Phase M.1 of the back-on-track plan (`docs/superpowers/specs/2026-04-20-back-on-track-design.md`). Core question: **does the rating → embedding → retrieval → director-injection chain actually work end-to-end, or is it silently broken?**

**Method:** Implemented `scripts/trace-director-prompt.ts` that mirrors `directListingScenes` and the production pipeline's director call. Ran live traces against two real targets and captured transcripts in `docs/traces/`.

- Listing trace: `dd552c89-98fd-4b6d-816c-1111323a208a` (10 photos, 11 scenes, 15 iterations, 6 rated — the most-iterated Phase 2.8 listing)
- Property trace: `6f508e16-096c-4a70-83cb-17b769838d61` ("2324", complete, 12 selected photos, 12 scenes, 7 ratings)

Pool-wide stats captured via direct Supabase queries.

## Verdict

**WORKING WITH GAPS.** The chain is wired end-to-end and producing real director signal in both directions (Lab listings → director + prod properties → director). Oliver's hypothesis that "the OpenAI embedding ML ended up being a failure" is **refuted** — 108 rated legacy Lab iterations + 7 prod scene ratings + 6 listing iterations are actively feeding the director via unified retrieval.

However: three meta-pathways on top of that chain have never fired in practice, and one data-coverage gap is partially suppressing the signal on production.

## Findings

### 1. Embedding coverage

| Layer | Embedded | Total | Rate |
|---|---|---|---|
| Phase 2.8 listing photos | 10 | 10 | 100% ✓ |
| Legacy Lab iterations (for retrieval pool) | — | 189 | pool works |
| Production scenes | 7 | 24 | 29% ✗ |

Production scene embedding runs via `embedScene()` at scene-insert time — but only scenes attached to a `scene_ratings` row get embedded by the backfill script. Unrated prod scenes remain un-embedded. **On the traced property: 7 of 12 scenes have embeddings.** The other 5 run through the pipeline with no retrieval-based in-context learning.

### 2. Retrieval — is it returning real exemplars?

Both traces returned populated blocks:

| Target | Recipes | Past winners | Past losers |
|---|---|---|---|
| Listing `dd552c89` | 5 | 5 | 3 |
| Property `6f508e16` | 2 | 5 | 3 |

Sample exemplar from the property trace: `5★ · aerial · drone_pull_back · runway` — an actual rated legacy Lab winner, surfaced by cosine distance against the property's aerial scene embedding. Sample recipe match: `other_push_in_260417_zybb` (room=other, movement=push_in, distance=0.266, applied 0× before).

**RPCs verified working:** `match_rated_examples`, `match_loser_examples`, `match_lab_recipes`. All return populated result sets.

### 3. Injection — does the director actually see this?

Yes. The full director user message on the property trace is 21,799 chars, of which the retrieval blocks (`VALIDATED RECIPE MATCH`, `PAST WINNERS`, `PAST LOSERS`) appear as expected, with content derived from real prior ratings. The blocks follow the canonical markers defined in `lib/prompt-lab.ts`. `DIRECTOR_SYSTEM` is 22,568 chars.

### 4. Signal pool sizes

| Source | Total rows | Rated (≥1★) |

| `scene_ratings` (production) | 7 | 7 (by definition) |

The signal pool is healthy — ~121 rated iterations feeding retrieval. Legacy Lab is the dominant contributor.

### 5. Lab→prod promotion pathway

**Never fired in practice.** Confirmed via:

```sql
SELECT COUNT(*) FROM lab_prompt_overrides WHERE is_active           → 0
SELECT COUNT(*) FROM lab_prompt_overrides WHERE promoted_to_prod_at IS NOT NULL → 0
SELECT COUNT(*) FROM prompt_revisions WHERE source = 'lab_override' → 0
```

The code path exists (`api/admin/prompt-lab/promote-to-prod.ts` + `resolveProductionPrompt` in `lib/prompts/resolve.ts`). Migration 016 added the `promoted_to_prod_at` columns. But no override has ever been promoted, and no active override exists.

**Implication:** production directors are using the baseline `DIRECTOR_SYSTEM` + unified retrieval (which works). Lab-tuned system-prompt rewrites have not been deployed to prod, despite the pathway being built.

This is not a broken pathway — it's an unused one. Oliver has never clicked "promote to prod" on an override. The feature shipped, and retrieval covers the in-context-learning half of the loop, but the system-prompt-rewrite half hasn't been exercised.

### 6. Rule-mining / proposals pathway

**Never activated in practice.** `lab_prompt_overrides WHERE is_active = true` returns 0 rows. Either no proposals were ever applied, or they were applied and later deactivated. In combination with finding 5 (zero overrides promoted), this whole meta-pathway (proposal → override → promotion) is theoretical today.

### 7. Recipe bloat

40 active recipes. Dedup was removed 2026-04-17. Not catastrophic at 40, but growing unbounded. Worth a soft cap per (room × movement) bucket to keep retrieval tight.

### 8. Schema drift spotted

The spec referred to `lab_prompt_overrides.promoted_at` (singular). The actual column is `promoted_to_prod_at`. Memory + spec reference will be corrected in M6.

## Recommended Phase M.2 actions

Ordered by leverage:

### Do first (high leverage, low effort)

1. **Back-fill production scene embeddings** — fix the 7/24 → 24/24 gap. Re-run `scripts/backfill-scene-embeddings.ts` with a widened filter that includes un-rated scenes (or add a `--all` flag). ~17 scenes × OpenAI text-embedding-3-small cost ≈ $0.00 effective. Unlocks retrieval on the majority of existing prod scenes.

2. **Commit the trace script as a permanent regression guard** — already done in commit `8cf3fd9`. Rerun after any change to director.ts / prompt-lab.ts / retrieval RPCs.

### Do second (capture-surface consolidation per the plan)

3. **Stop writing deprecated capture fields** — `iteration.tags` (superseded by `rating_reasons`), `iteration.refinement_instruction` (superseded by scene-level `refinement_notes`), `iteration.chat_messages` (superseded by scene-level `chat_messages`). Preserve read paths for historical retrieval (per directive Q6-B).

4. **Remove confirmed dead code** — `match_lab_iterations` RPC (unused per project-state), `lib/prompts/prompt-qa.ts` + `runPreflightQA` body. Low risk, small diff.

### Do third (pathway disposition)

5. **Lab→prod promotion: KEEP, surface more prominently.** The pathway works in code; it just hasn't been used. Consider adding a UI nudge on the Proposals page when a Lab override crosses the readiness threshold (`lab_prompt_override_readiness` view), so Oliver doesn't miss eligible promotions. Do NOT delete — the feature is ready for its first real use.

6. **Rule-mining proposals: KEEP AS-IS, flag "experimental" in UI.** Zero proposals applied is a usage signal, not a broken-code signal. Leave the tables + endpoints.

7. **Recipe soft cap** — when there are >10 active recipes per (room_type × camera_movement) bucket, mark the oldest by `promoted_at` and lowest-rated ones as `status='archived'`. Not urgent at current counts.

### Schema hygiene

8. **Correct `promoted_at` → `promoted_to_prod_at` references** in memory + any internal docs (the spec I wrote yesterday got this wrong).

## What NOT to do

- **Don't panic-rewrite the learning loop.** It works. The meta-pathways (Lab→prod promotion, rule-mining) are unused but not broken.
- **Don't delete legacy Lab data.** The 108 rated legacy iterations are by far the largest training signal we have, and they're flowing into every new director run via `match_rated_examples`.
- **Don't re-enable recipe dedup.** The user chose to remove it; 40 recipes is fine. Soft cap is the right control at higher volumes.
