# Phase 2.6 — Reverse-in-Assembly Wiring Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let the director flag scenes that want a "pullout feel" with `reverse_in_assembly: true`, and have the assembly stage actually reverse the rendered push_in clip before handing the final video to Shotstack. Ships the second half of the pullout-removal decision: Phase 2.5 dropped pullouts from AI generation; Phase 2.6 puts the post-production reversal in place so we don't lose the pullout aesthetic.

**Architecture:** Add `reverse_in_assembly BOOLEAN` to scenes and prompt_lab_iterations. Director system prompt learns to set it true for "classic drone pullback opening," "pull-out kitchen reveal," etc. Build a `lib/services/video-reverse.ts` with a `reverseClipUrl(url)` function that wraps Cloudinary's fetch-mode `e_reverse` URL transformation (free tier, no upload step — Cloudinary fetches the clip URL, serves a reversed variant). The assembly pipeline calls `reverseClipUrl` for any clip flagged `reverse_in_assembly=true` before building the Shotstack timeline, and Shotstack sees a pre-reversed MP4 URL. Lab iteration previews apply the same logic so Oliver sees the final look when rating.

**Tech Stack:** TypeScript (ESM), Vercel Serverless Functions (Node 20), Supabase Postgres, Cloudinary URL transformations (free tier, fetch mode — no account upload required for MVP; paid tier handles scale later), Shotstack Edit API v1 (unchanged, receives reversed URLs).

**Scope boundaries (do NOT build in this plan):**

- No ffmpeg-in-Vercel-Sandbox path (Cloudinary handles it; ffmpeg-based approach is a deferred fallback)
- No change to Shotstack integration (Shotstack still receives plain MP4 URLs; reversal happens upstream)
- No UI for toggling `reverse_in_assembly` per-scene in Lab (director sets the flag automatically; manual override deferred)
- No director changes beyond allowing + documenting the new field
- No per-aspect-ratio variant reversal — we reverse once, Shotstack uses the same reversed URL across 9:16 and 16:9 outputs

Create `lib/services/video-reverse.ts`:

```ts
// Reverses a clip URL via Cloudinary's fetch-mode URL transformation.
// Free tier requires only CLOUDINARY_CLOUD_NAME (no API key for fetch
// mode). Cloudinary downloads the remote URL, applies e_reverse, and
// serves the result under a deterministic URL that any MP4 consumer
// (incl. Shotstack) can treat as a plain remote video.
//
// Phase 2.6: AI generates inward motion (push_in, drone_push_in) only.
// When the director flags a scene reverse_in_assembly=true, the
// pipeline calls reverseClipUrl before handing the clip to Shotstack.

const CLOUDINARY_BASE = "https://res.cloudinary.com";
const REVERSE_PATH = "/video/fetch/e_reverse/";

export function isReversible(url: string): boolean {
  if (typeof url !== "string" || url.length === 0) return false;
  try {
    const u = new URL(url);
    return u.protocol === "https:";
  } catch {
    return false;
  }
}

// Returns the Cloudinary-wrapped reversed URL. Idempotent: if the input
// is already a Cloudinary reversed URL, returns it unchanged.
export async function reverseClipUrl(url: string): Promise<string> {
  const cloudName = process.env.CLOUDINARY_CLOUD_NAME;
  if (!cloudName) {
    throw new Error("CLOUDINARY_CLOUD_NAME is not set — cannot reverse clip URL");
  }
  if (!isReversible(url)) {
    throw new Error(`URL is not reversible (must be https): ${url}`);
  }
  const prefix = `${CLOUDINARY_BASE}/${cloudName}${REVERSE_PATH}`;
  if (url.startsWith(prefix)) return url;
  return `${prefix}${encodeURIComponent(url)}`;
}
```

```ts
import { reverseClipUrl } from "./services/video-reverse.js";
```

(Adjust relative path to match the file you're editing.)

Add a helper near the timeline-builder entry point:

```ts
async function resolveClipUrlForAssembly(
  scene: { clip_url: string | null; reverse_in_assembly: boolean }
): Promise<string | null> {
  if (!scene.clip_url) return null;
  if (!scene.reverse_in_assembly) return scene.clip_url;
  try {
    return await reverseClipUrl(scene.clip_url);
  } catch (err) {
    // If Cloudinary is misconfigured or the URL can't be wrapped, fall
    // back to the original clip rather than failing the whole render.
    console.error(`[assembly] reverseClipUrl failed for scene ${(scene as { scene_number?: number }).scene_number ?? "?"}:`, err);
    return scene.clip_url;
  }
}
```

Replace the existing direct-use of `scene.clip_url` in the timeline builder with:

```ts
const urlForTimeline = await resolveClipUrlForAssembly(scene);
if (!urlForTimeline) continue; // or throw, matching existing error semantics
```

Use `Promise.all` to parallelize the per-scene lookups so total assembly latency is unchanged (Cloudinary fetch-mode URLs resolve instantly — they're just string wrapping; the actual fetch happens inside Shotstack).
